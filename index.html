Similarities

  • Both extract 10 frames per second → 190 frames total from the 19s video
  • Both output JPEG image files
  • Both produce visually identical frames — no perceptible difference to the eye, as visible in the comparison above

Differences

  • File size: OpenCV produces ~33 KB per frame vs MoviePy's ~15 KB (~54% smaller). This is because OpenCV defaults to JPEG quality ≈ 95 (low compression), while MoviePy defaults to ≈ 85 (higher compression). The visual result is the same because the quality difference is below perceptual threshold for this type of footage.
  • Timing precision: OpenCV captures the nearest real frame in the video stream (e.g. 0.12s, 0.21s, 0.33s), while MoviePy targets exact decimal intervals (0.10s, 0.20s, 0.30s). Both cover the full video.
  • Code simplicity: MoviePy requires fewer lines and is easier to read.
The frames look identical because the compression difference (quality 95 vs 85) is imperceptible for this footage. The ~54% size reduction is purely an encoding choice, not a quality trade-off.

Preferred method

MoviePy — more concise code and output files are ~54% smaller with no visual quality loss.

What we learned

We observed how different libraries apply different default JPEG compression settings during frame extraction. OpenCV produced images averaging about 33 KB per frame, while MoviePy produced images averaging about 15 KB per frame. Despite the large difference in file size, the extracted frames appeared visually identical because both libraries captured the same video content and the compression difference was below the level noticeable to the human eye.

We also observed that OpenCV extracts frames based on the actual frame positions stored in the video stream, whereas MoviePy samples frames using exact timestamps. Although the extraction methods differ internally, both approaches successfully represented the original video and produced nearly identical visual results.

Comparing the extracted frames with the original video helped us understand that a video is fundamentally a sequence of still images displayed over time. By extracting frames, we could directly observe how motion in the video is created from individual images.

Method → Observation → Conclusion

Method: OpenCV and MoviePy extracted 190 JPEG frames from the same video.

Observation: MoviePy generated files that were approximately 54% smaller while preserving visual quality.

Conclusion: Different default compression settings can significantly reduce storage requirements without introducing perceptible visual differences.

How were bitrate and duration measured?
We used the ffprobe utility included with FFmpeg to inspect the exported MP3 files:

ffprobe zoo_ffmpeg.mp3
ffprobe zoo_moviepy.mp3

The output reported:
  • FFmpeg: Duration = 18.94 s, Bitrate = 64 kb/s, 48 kHz mono audio
  • MoviePy: Duration = 18.96 s, Bitrate = 128 kb/s, 44.1 kHz stereo audio
These values were used to populate the comparison table above. The nearly identical durations confirm that both methods extracted the complete audio track from the original video, while the bitrate difference explains the approximately 2× larger file size produced by MoviePy.

Method → Observation → Conclusion

Method: ffprobe was used to inspect the exported audio metadata.

Observation: MoviePy generated stereo 128 kbps audio, while FFmpeg generated mono 64 kbps audio.

Conclusion: Encoding parameters such as bitrate and channel configuration directly influence file size without necessarily affecting human perception.

Similarities

  • Both take the same video file as input and output an MP3
  • Both produce audio that sounds identical for this clip — the bitrate difference is noticeable only in high-frequency detail under careful listening
  • Both are invoked from the terminal with a similar command structure

Differences

FFmpeg

  • Requires separate system installation + manual PATH config on macOS
  • Defaults to 64 kbps → smaller file (148 KB)
  • Highly configurable (bitrate, codec, channels, sample rate)
  • Single terminal command, no Python needed

MoviePy

  • Installed via pip — no extra system setup
  • Defaults to 128 kbps → larger file (297 KB), higher fidelity
  • Pure Python API — integrates directly into scripts
  • Internally calls FFmpeg under the hood
Why the size difference? FFmpeg defaulted to 64 kbps encoding while MoviePy defaulted to 128 kbps. Higher bitrate means more data per second of audio → larger file but better preservation of high-frequency detail. For typical speech or ambient sound like this zoo clip, 64 kbps is already sufficient quality. MoviePy is actually a Python wrapper around FFmpeg — both use the same encoding engine underneath, which is why the audio sounds essentially the same despite the bitrate difference.

Preferred method

MoviePy — simpler setup (especially on macOS where FFmpeg requires manual installation) and integrates into Python scripts without switching tools.

What we learned

Using FFmpeg's ffprobe tool allowed us to inspect the technical properties of the extracted audio files, including duration, bitrate, sample rate, and channel configuration. The FFmpeg output showed a duration of 18.94 seconds and a bitrate of 64 kb/s, while the MoviePy output showed a duration of 18.96 seconds and a bitrate of 128 kb/s.

We learned that bitrate has a direct impact on file size. Since MoviePy encoded the audio at 128 kb/s, the resulting file was approximately twice as large as the FFmpeg version encoded at 64 kb/s. However, when listening to both files, the audible difference was minimal because the original clip mainly contains ambient zoo sounds and does not require a high bitrate to preserve perceptual quality.

This observation showed us that larger files do not always produce noticeably better audio. The encoding settings affect storage requirements much more than perceived quality for simple audio content.

What we learned

Creating a video from a static image demonstrated how video and audio streams can be combined into a single multimedia file. We learned that the image clip duration must match the audio duration to ensure proper synchronization throughout playback.

We also observed the relationship between frame rate and file size. Since the image never changes, using 1 fps is sufficient because additional frames would contain identical visual information. As a result, the encoder only needed to store about 19 frames for the entire video instead of hundreds of frames at a traditional frame rate such as 30 fps.

This directly explains why the generated video remained visually identical while requiring significantly less storage space and encoding time. The method (using a low frame rate for a static image) directly produced the observed result (a small output file).

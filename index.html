Question 1 — Extract Frames from Video

Both OpenCV and MoviePy were used to extract frames from a 19-second zoo video at 10 fps, producing 190 frames each.

Outcome

OpenCV190JPEG~33 KB
MoviePy190JPEG~15 KB

Visual comparison — select a timestamp

File size per frame

OpenCV MoviePy

Similarities

  • Both extract 10 frames per second → 190 frames total from the 19s video
  • Both output JPEG image files
  • Both produce visually identical frames — no perceptible difference to the eye, as visible in the comparison above

Differences

  • File size: OpenCV produces ~33 KB per frame vs MoviePy's ~15 KB (~54% smaller). This is because OpenCV defaults to JPEG quality ≈ 95 (low compression), while MoviePy defaults to ≈ 85 (higher compression). The visual result is the same because the quality difference is below perceptual threshold for this type of footage.
  • Timing precision: OpenCV captures the nearest real frame in the video stream (e.g. 0.12s, 0.21s, 0.33s), while MoviePy targets exact decimal intervals (0.10s, 0.20s, 0.30s). Both cover the full video.
  • Code simplicity: MoviePy requires fewer lines and is easier to read.
The frames look identical because the compression difference (quality 95 vs 85) is imperceptible for this footage. The ~54% size reduction is purely an encoding choice, not a quality trade-off.

Preferred method

MoviePy — more concise code and output files are ~54% smaller with no visual quality loss.

What we learned

We observed how different libraries apply different default JPEG quality settings, directly affecting file size without affecting visual quality. We also saw how OpenCV aligns to actual frame boundaries in the video stream, while MoviePy interpolates at exact time intervals — both valid strategies depending on the use case.

Question 2 — Extract Audio from Video

FFmpeg and MoviePy were both used to extract the audio track from the zoo video as an MP3 file.

Similarities

  • Both take the same video file as input and output an MP3
  • Both produce audio that sounds identical for this clip — the bitrate difference is noticeable only in high-frequency detail under careful listening
  • Both are invoked from the terminal with a similar command structure

Differences

FFmpeg

  • Requires separate system installation + manual PATH config on macOS
  • Defaults to 64 kbps → smaller file (148 KB)
  • Highly configurable (bitrate, codec, channels, sample rate)
  • Single terminal command, no Python needed

MoviePy

  • Installed via pip — no extra system setup
  • Defaults to 128 kbps → larger file (297 KB), higher fidelity
  • Pure Python API — integrates directly into scripts
  • Internally calls FFmpeg under the hood
Why the size difference? FFmpeg defaulted to 64 kbps encoding while MoviePy defaulted to 128 kbps. Higher bitrate means more data per second of audio → larger file but better preservation of high-frequency detail. For typical speech or ambient sound like this zoo clip, 64 kbps is already sufficient quality. MoviePy is actually a Python wrapper around FFmpeg — both use the same encoding engine underneath, which is why the audio sounds essentially the same despite the bitrate difference.

Preferred method

MoviePy — simpler setup (especially on macOS where FFmpeg requires manual installation) and integrates into Python scripts without switching tools.

What we learned

We learned that audio file size is directly determined by bitrate: MoviePy's 128 kbps output is exactly double the size of FFmpeg's 64 kbps output because it stores twice the data per second. We also discovered that MoviePy is a Python wrapper around FFmpeg — understanding this helped us debug compatibility issues we encountered. Finally, we learned to set up a development environment on macOS from scratch, adapting Windows-based instructions to use Homebrew instead of manual PATH configuration.

Question 3 — Combine Image with Audio

A static image was combined with the extracted audio track to produce a video file.

Outcome

PropertyValue
Output formatMPEG-4 (.mp4)
File size704 KB
FPS used1 fps
DurationMatched to audio clip (~19s)

What we learned

Setting fps=1 for a static image video is a coding best practice: since the image never changes, only one frame per second needs to be encoded instead of 24 or 30. This directly reduces both file size and encoding time, with no perceptible quality loss — the video looks identical because a repeated static frame carries no new information.

We also learned how to synchronize clip durations: the image clip's duration must be explicitly set to match the audio duration, otherwise MoviePy either cuts the audio short or runs the video silent at the end. This reflects how video containers work — the video and audio tracks are separate streams that must be aligned.

The relationship between method and result is direct: fps=1 means the encoder stores ~19 frames total instead of ~570 (at 30 fps), which explains why a static-image video with audio is only 704 KB despite being 19 seconds long.

Bonus — Custom Multimedia Applications

Two additional tools built on top of the project's core libraries.

1. Audio Trimmer (CLI)

Command-line script that accepts a start time and end time in seconds as arguments and saves only that portion as a new MP3 file.

Usagepython audio_trimmer.py <start> <end>
Examplepython audio_trimmer.py 2 8 → exports seconds 2–8
Outputzoo_trimmed_2s_to_8s.mp3 (96.8 KB, 6-second clip)
LibraryMoviePy — AudioFileClip.subclip()

2. GUI Frame & Audio Extractor

Desktop application with a graphical interface built using Tkinter. The user selects any video file through a file dialog, then extracts all frames as PNG or the audio track as WAV, with a real-time progress bar during frame extraction.

InterfaceTkinter (Python standard library)
Frame outputPNG files saved to extracted_frames_GUI/
Audio outputWAV (PCM 16-bit) saved to extracted_audio_GUI/
Extra featureReal-time progress bar — updates every 10 frames
Both tools were built from scratch using only the same libraries used in the main project questions (OpenCV, MoviePy, Tkinter). No additional dependencies.
